# 代码说明（Round 0）

## 目前 `main.py` 做了什么

当前 `main.py` 只做一件最小事情：

1. 定义 `main()` 函数；
2. 在 `main()` 中打印 `novel-demo initialized`；
3. 使用 `if __name__ == "__main__":` 作为标准入口来调用 `main()`。

这意味着当前仓库已经具备“可执行入口”，并且能让学习者快速验证环境是否正常。

## 后续会扩展什么

后续轮次将逐步扩展到：

1. **读取配置**：通过 `.env` 与 `python-dotenv` 读取 API 配置；
2. **输入处理**：从 `inputs/` 读取故事创意文本；
3. **Skill 装配**：把 `skills/` 中的模板内容拼接为提示词；
4. **API 调用**：向模型发送请求并接收返回结果；
5. **输出落盘**：把生成结果写入 `outputs/`，形成可追踪产物；
6. **工程完善**：加入日志、错误处理与更多教学文档。

## 为什么 Round 0 只做骨架

这是教学项目的关键策略：

- 先把目录、约定、入口统一好；
- 再按轮次逐步增加复杂度；
- 每次只引入少量新概念，便于理解与回顾。

这种方式可以显著降低初学阶段的认知负担。

## Round 1 补充说明：什么是 skill 文件

`skills/` 目录下的文件可以理解为“写作任务说明书”。

你可以把它想成给模型的一份固定规则：
- 这次任务要做什么；
- 输出要用什么结构；
- 哪些行为要避免。

例如本项目的 `skills/novel_outline.md`，就规定了小说大纲必须按固定章节输出，并明确了语气和边界。

## skill 如何影响模型输出

当模型拿到同一个用户输入时：
- 如果没有 skill，输出可能每次都不一样，结构也可能混乱；
- 如果有 skill，模型会更容易按统一格式整理内容。

所以，skill 的核心价值是：
- 提高输出稳定性；
- 降低“想到哪写到哪”的随机性；
- 让后续流程（比如保存、复用、二次加工）更容易。

## 当前状态与后续计划

目前 `main.py` 还没有真正去读取 `skills/novel_outline.md`。

但在后续轮次中，会把这个 skill 文件读入程序，并作为 **system prompt（系统提示词）** 使用。

这里的“系统提示词”可以简单理解为：
- 在正式处理用户内容之前，先告诉模型“你要遵守哪些固定规则”。

这样就能把文档里写好的规范，真正变成程序执行时的行为约束。

## Round 2：API 调用流程说明

这一轮的目标是打通“最小可运行闭环”：

- 读取配置
- 读取 skill
- 接收用户输入
- 调用模型
- 打印结果

下面按零基础视角解释每一步。

### 1. `.env` 的作用

`.env` 可以理解为“本地配置文件”。
它把账号密钥、模型名称、接口地址这类会变化的信息放在代码外面管理。

好处是：
- 改配置时不用改 Python 代码；
- 不同同学可以用自己的密钥与模型；
- 避免把敏感信息硬编码进脚本。

### 2. API Key 的作用

`OPENAI_API_KEY` 是访问模型服务的身份凭证。

你可以把它理解为“门禁卡”：
- 没有卡，服务端不知道你是谁；
- 卡无效，服务端会拒绝请求；
- 卡有效，才能进行计费与权限校验。

所以程序在启动时会先检查它是否为空，避免跑到一半才报错。

### 3. `base_url` 的作用

`OPENAI_BASE_URL` 决定请求发往哪里。

- 留空：走 OpenAI 官方默认地址；
- 填写：走你配置的聚合平台（前提是它兼容 OpenAI 接口格式）。

这就是“同一份代码可切换不同服务商”的关键点。

### 4. `model` 的作用

`OPENAI_MODEL` 决定“调用哪一个模型”。

即使 API Key 正确，如果模型名写错，也会调用失败。
因此程序也会在启动阶段检查模型名是否为空。

### 5. 为什么把 skill 文件作为 `system prompt`

`skills/novel_outline.md` 里写的是“任务规则”和“输出结构”。
这类内容最适合放在 `system` 角色中，因为它代表全局指令。

这样做的效果是：
- 模型先理解固定规则；
- 再处理用户创意；
- 输出更稳定、更接近项目要求。

### 6. 为什么把用户创意作为 `user message`

用户在终端输入的是“本次任务的具体需求”，属于一次性内容。

所以它应该放在 `user` 角色，让模型知道：
- 哪些是固定规则（system）；
- 哪些是当前问题（user）。

这种角色拆分是对话式 API 的基础用法。

### 7. 什么是 OpenAI 兼容接口

“OpenAI 兼容接口”是指：
- 第三方平台按 OpenAI 的请求格式来实现接口；
- 客户端代码基本不需要大改；
- 通常只改 `api_key`、`base_url`、`model` 即可切换。

因此本项目选用 OpenAI 官方 Python SDK，学习成本更低，也更通用。

### 8. 为什么要加 `try/except`

API 调用是网络操作，可能出现：
- 网络不通；
- Key 错误；
- 模型名错误；
- 平台临时故障。

如果不做异常处理，程序会直接抛出长报错栈，对初学者不友好。

加上 `try/except` 后可以：
- 先输出“API 调用失败”的清晰提示；
- 再附带异常信息，帮助定位问题；
- 保证失败时也有可理解的反馈。
