# 代码说明（Round 0）

## 目前 `main.py` 做了什么

当前 `main.py` 只做一件最小事情：

1. 定义 `main()` 函数；
2. 在 `main()` 中打印 `novel-demo initialized`；
3. 使用 `if __name__ == "__main__":` 作为标准入口来调用 `main()`。

这意味着当前仓库已经具备“可执行入口”，并且能让学习者快速验证环境是否正常。

## 后续会扩展什么

后续轮次将逐步扩展到：

1. **读取配置**：通过 `.env` 与 `python-dotenv` 读取 API 配置；
2. **输入处理**：从 `inputs/` 读取故事创意文本；
3. **Skill 装配**：把 `skills/` 中的模板内容拼接为提示词；
4. **API 调用**：向模型发送请求并接收返回结果；
5. **输出落盘**：把生成结果写入 `outputs/`，形成可追踪产物；
6. **工程完善**：加入日志、错误处理与更多教学文档。

## 为什么 Round 0 只做骨架

这是教学项目的关键策略：

- 先把目录、约定、入口统一好；
- 再按轮次逐步增加复杂度；
- 每次只引入少量新概念，便于理解与回顾。

这种方式可以显著降低初学阶段的认知负担。

## Round 1 补充说明：什么是 skill 文件

`skills/` 目录下的文件可以理解为“写作任务说明书”。

你可以把它想成给模型的一份固定规则：
- 这次任务要做什么；
- 输出要用什么结构；
- 哪些行为要避免。

例如本项目的 `skills/novel_outline.md`，就规定了小说大纲必须按固定章节输出，并明确了语气和边界。

## skill 如何影响模型输出

当模型拿到同一个用户输入时：
- 如果没有 skill，输出可能每次都不一样，结构也可能混乱；
- 如果有 skill，模型会更容易按统一格式整理内容。

所以，skill 的核心价值是：
- 提高输出稳定性；
- 降低“想到哪写到哪”的随机性；
- 让后续流程（比如保存、复用、二次加工）更容易。

## 当前状态与后续计划

目前 `main.py` 还没有真正去读取 `skills/novel_outline.md`。

但在后续轮次中，会把这个 skill 文件读入程序，并作为 **system prompt（系统提示词）** 使用。

这里的“系统提示词”可以简单理解为：
- 在正式处理用户内容之前，先告诉模型“你要遵守哪些固定规则”。

这样就能把文档里写好的规范，真正变成程序执行时的行为约束。
